# AI ChatRobot 项目架构决策

## 请求生命周期

一次聊天请求依次经过 Router、ChatService、Memory、PromptManager、Provider 和大模型。Router 只处理 HTTP 协议；ChatService 负责工作流编排；Provider 只负责把应用领域对象翻译为模型厂商协议。各层保持单一职责，新增能力时通过插入生命周期节点完成演进。

## Memory 的边界

Memory 只保存用户可见的对话事实，包括用户消息和模型最终回复。System Prompt 属于应用配置，不写入 Memory。模型产生的工具调用、工具参数和工具执行结果是一次请求内部的临时轨迹，也不写入 Memory，避免底层执行细节污染后续对话上下文并持续消耗 Token。

当前 ChatMemory 是进程运行期内存，只能在服务运行期间跨请求保留对话；服务重启后记录会丢失，它不属于数据库持久化。永久会话存储是后续独立能力，不能与当前实现混淆。

如果未来需要排查问题或满足合规要求，工具轨迹应进入独立的 Tool Execution Log。审计日志和聊天记忆具有不同的保留周期、查询方式与访问权限，不能混用同一个存储模型。

## PromptManager 的职责

PromptManager 将 System Prompt 与最近聊天历史组合成发送给模型的最终消息列表。它负责上下文裁剪，但不保存聊天记录，也不调用模型。当前版本按消息条数限制历史；未来可以替换为 Token 预算和历史摘要策略，而不修改 Router、Memory 或 Provider。

## Tool Calling 的安全边界

模型生成的工具名称和参数始终是不可信输入。系统只允许执行 ToolRegistry 中显式注册的工具，不使用 eval，不允许模型指定 Python 模块、函数或系统命令。参数错误会转换为结构化工具结果；未知内部异常不会原样暴露给模型。

单次请求最多执行有限轮工具调用，防止模型重复请求形成无限循环。当前工具轨迹只存在于请求内存中，最终用户可见回答才会进入 Memory。

## Knowledge RAG 的定位

知识库检索通过 search_knowledge_base 工具按需触发。普通聊天不强制检索；涉及项目私有知识、架构决策或内部文档时，模型应调用知识工具。检索结果必须携带 source 和 chunk_id，最终回答应引用这些真实来源，不能伪造不存在的文档。
